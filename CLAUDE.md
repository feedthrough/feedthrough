# Feedthrough тАФ Claude Code instructions

## What this is

A browser debug bridge exposed as an MCP server. `@feedthrough/core` injects into any web page
and intercepts console output, fetch, and XHR. `@feedthrough/mcp` runs locally and exposes
everything as MCP tools over stdio so any MCP-compatible AI agent (Claude Code, Cursor, etc.)
can inspect and drive the page in real time.

The key differentiator: embedded inside the page (not an external Puppeteer/CDP observer), so
it works in any browser and has access to framework internals.

## TypeScript config

- Browser packages (`core`): `module: ESNext`, `moduleResolution: Bundler`, `lib: [DOM, ES2020]`
- Node packages (`mcp`, `cypress`, `vite`, `webpack`): `module: NodeNext`,
  `moduleResolution: NodeNext` тАФ relative imports need `.js` extensions
- All packages extend `tsconfig.base.json` at the repo root

## Build quirks

**esbuild CLI is broken in pnpm** тАФ the pnpm cmd-shim wrapper calls `node <native-binary>` which
fails. Always use the esbuild JS API instead. See `packages/core/scripts/build-iife.mjs` for the
pattern.

**Cypress adapter has a generated file** тАФ `packages/cypress/src/generated/bundle.ts` is
auto-generated by `scripts/embed-bundle.mjs` during `prebuild`. It embeds the core IIFE as a
string constant. It is gitignored; never edit it manually or commit it.

**Build order matters** тАФ `@feedthrough/cypress` depends on `@feedthrough/core`'s IIFE output,
so core must be fully built before cypress. `pnpm build` (root) handles this automatically via
workspace dependency ordering.

## Package details

### @feedthrough/core

- `src/transport.ts` тАФ WebSocket client with reconnect and outbound queue
- `src/interceptors/console.ts` тАФ patches every console method (log/warn/error/info/debug plus
  dir, table, assert, trace, count/countReset, time/timeEnd/timeLog, group/groupCollapsed/
  groupEnd, clear); rich methods carry a `method` field, trace/assert add `stack`
- `src/interceptors/network.ts` тАФ wraps `window.fetch` and `XMLHttpRequest`
- `src/commands.ts` тАФ dispatches incoming commands: click, fill, hover, inspect, query_dom, etc.
- `src/bridge.ts` тАФ `FeedthroughBridge` class, wires everything together
- `src/browser.ts` тАФ auto-init entry point for the IIFE bundle; reads config from
  `window.__feedthroughOptions`
- `dist/feedthrough.iife.js` тАФ self-contained 9 KB bundle for direct page injection

Public API: `init(options?)` (convenience) and `FeedthroughBridge` class.

### @feedthrough/mcp

- `src/bridge-client.ts` тАФ WebSocket server on `:8765`, manages browser connection,
  maps command IDs to pending promises with 10 s timeouts
- `src/server.ts` тАФ `McpServer` with sixteen tools registered via `registerTool` (not the
  deprecated `tool()`). Uses MCP SDK 1.29.0 / Zod v4.
- All logging goes to `process.stderr` тАФ stdout is reserved for the MCP stdio protocol
- Port override: `FEEDTHROUGH_PORT` env var

### @feedthrough/cypress

- `src/support.ts` тАФ `setupFeedthrough(options?)` hooks `window:before:load` and calls
  `win.eval(bridgeBundle)` to run the bridge in the AUT window scope
- User setup: `import { setupFeedthrough } from "@feedthrough/cypress"` in
  `cypress/support/e2e.ts`, then call `setupFeedthrough()`

### website (Astro)

- `src/pages/index.astro` тАФ pure component assembly, no markup or styles of its own
- `src/components/` тАФ one `.astro` file per section, scoped styles
- `src/styles/global.css` тАФ CSS variables, reset, shared utilities
- `src/layouts/Layout.astro` тАФ HTML shell with fonts, meta, scroll observer
- Deploy: GitHub Actions workflow (`.github/workflows/deploy-website.yml`), manual trigger only
  until repo goes public тАФ then flip to `push: branches: [main]`

### website/demos (demo-video harness)

Separate Vite + React app (`@feedthrough/demos`, in the `website/demos` workspace package) whose
only job is to generate the landing-page clips. It's **not** built by Astro тАФ Astro only touches
`website/src` and copies `website/public`. The clips are **scripted reenactments**: the harness
plays a canned agent transcript while applying the *real* DOM effects to a real buggy app;
Playwright records the viewport, ffmpeg encodes to `website/public/demo-*.{mp4,webm,gif}`.

- Edit the scripts in `src/timelines.ts`; the buggy apps are in `src/demos/`.
- Regenerate: `pnpm --filter @feedthrough/demos record && тАж encode`, then rebuild the website so
  the static build picks up the new files. Committed output lives in `website/public/`.
- Shown on the site via `website/src/components/Demos.astro` (tabbed video section after the Hero).
- Full detail (timing/sizing knobs, the white-frame trim) is in `website/demos/README.md`.

### examples/react-app

Contains three deliberate bugs for testing the MCP diagnostic workflow:
1. Counter increments state by 2 but logs +1 тАФ detectable via `click` + `get_console_logs` +
   `inspect_element`
2. Team search is case-sensitive (missing `.toLowerCase()`) тАФ detectable via `fill` + `query_dom`
3. Activity feed fetches `/api/events` (wrong URL, 404), error swallowed silently тАФ detectable
   via `get_network_requests` + `get_console_logs`

## MCP tools (all sixteen)

Observe: `get_instructions`, `connection_status`, `get_page_info`, `get_console_logs`

Interact: `click`, `fill`, `hover`, `press_key`
Live-edit (preview, not saved to source): `set_style`, `set_attribute`, `set_text`,
`reset_overrides`

## Documentation map тАФ keep these in sync

The tool list and other user-facing facts are duplicated across several places. When you
**add, remove, or change an MCP tool** (or its inputs), update ALL of these in the same change тАФ
they drift silently otherwise:

- **Core** тАФ `packages/core/src/types.ts` (the `Command` union) and `commands.ts` (dispatch +
  implementation); for console/network behaviour, the relevant interceptor.
- **MCP server** тАФ `packages/mcp/src/server.ts` (`registerTool` call + its description/inputSchema).
- **Root `README.md`** тАФ the `## MCP tools` table.
- **`packages/mcp/README.md`** тАФ the `## Tools` table (has an Input column).
- **Website tool data** тАФ `website/src/data/tools.ts`. Single source of truth for the site;
  drives BOTH the homepage cards (`components/Tools.astro`) and the `/tools` reference page
  (`pages/tools.astro`). Edit here, not in the components.
- **Tool count** тАФ the `<h2>` in `website/src/components/Tools.astro` AND the тАЬN MCP toolsтАЭ
  line in `website/src/components/HowItWorks.astro`, AND the `## MCP tools (all N)` heading
  here, AND the тАЬтАжwith N toolsтАЭ bullet under Package details тЖТ `src/server.ts` above.
- **Tests** тАФ for tool/wire-protocol behaviour, add/adjust a case in
  `examples/playwright-app/tests/bridge-protocol.spec.ts`. For `BridgeClient` internals that the
  stand-in server can't reach (startup/bind errors, connection bookkeeping), use the `node:test`
  unit tests in `packages/mcp/test/` (`pnpm --filter @feedthrough/mcp test`).

Where other things live: per-adapter install/usage is in each `packages/*/README.md`;
landing-page copy is in `website/src/components/*.astro`; long-term plans and decisions are in
the session auto-memory, not the repo.

WebSocket server binds to `127.0.0.1` only and validates the `Origin` header (localhost origins
only) тАФ v1 is explicitly local-only.
